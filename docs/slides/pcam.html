PCAM
並列アルゴリズム設計手法

Partitioning → Communication → Agglomeration → Mapping

分割 通信 凝集 割当

出典：Ian Foster Designing and Building Parallel Programs（Chapter 2）
NotebookLM の原典ソースに基づく一次情報

概要 PCAM とは何か

  • 並列アルゴリズムを設計するための体系的・探索的なフレームワーク。単純なレシピではなく、設計の選択肢を最大化し代替案を評価する仕組み。
  • 核となる哲学：「抽象的・機種非依存の課題」と「具体的・機種依存の課題」を意図的に切り離す。
  • 悪い設計選択からの後戻り（backtracking）コストを下げることを狙う。
◀ 抽象・機種非依存（並列性を暴き出す）具体・機種依存（性能を作り込む）▶
P並列性の発見
C並列性の発見
A効率・局所性
M効率・局所性
前半2段階＝並列性とスケーラビリティを暴く / 後半2段階＝ハードウェア固有の性能課題に焦点

  • P 計算とデータを細粒度の小タスクに分解。プロセッサ数などの制約は無視し、並列実行の機会を発見。
  • C タスク間で必要なデータ移動（情報フロー）を規定する。
  • A 実際の計算機で効率よく動くよう、タスクを合体（粗粒度化）し局所性を高める。
  • M 各タスクを物理プロセッサに割り当てる。利用率最大化と通信最小化を両立させる。

なぜこの順序なのか

  • P が最初 実際の制約を無視することで、並列実行の機会を最大限に暴き出す。早期にスケーラビリティを制限したり、代替戦略を先入観で捨てたりしない。
  • C が2番目 P のタスクは独立に実行できない。機種非依存の抽象アルゴリズムを、ここで完成させる。
  • A が3番目 「抽象から具体への移行」。細粒度のままでは、タスク生成コストや大量の小メッセージで実機では非効率。理想構造が判明した後に合体させる。
  • M が最後 最も機種依存性が高い。汎用の割当機構は存在せず複雑なトレードオフを要するため、最後に置きハードウェア変更にアルゴリズムを書き直さず対応。

Partitioning 分割の目的

  • 計算とデータの両方を小タスクに分解し、並列実行の機会を暴き出す。狙うのは fine-grained（細粒度）。
  • この段階ではプロセッサ数などの実際的制約を意図的に無視する。
  • データ・計算の冗長な複製を避け、後続ステージへの最大限の柔軟性を確保する。
"programmers most commonly first focus on the data associated with a problem, then determine an appropriate partition for the data, and finally work out how to associate computation with data. This partitioning technique is termed domain decomposition."

Partitioning 2つの分割技法

Domain decomposition
（データ中心分割）

  • まずデータを均等な小片に分割し、各片に計算を割り当てる。
  • 例：大気モデル 3次元グリッドの1格子点＝1タスク。各タスクが点の状態保持と更新を担う（Fig 2.2）。
  • 3次元分割が最も柔軟で、設計初期に採用。

Functional decomposition
（機能中心分割）

  • まず計算を互いに素なタスクに分け、必要なデータを後から検討。
  • 例：探索木（VLSIフロアプラン）。再帰呼び出しごとにタスクを動的生成。
  • 例：気候モデルを大気・海洋・水文・地表のモジュールに分割。

両者は相補的。同一問題に両方適用し、複数の並列アルゴリズム案を得ることもできる。

Partitioning 設計チェックリスト

§2.2.3 — 原則すべてに「Yes」と答えられること。「No」があれば設計に欠陥の可能性。

  • タスク数はプロセッサ数より少なくとも1桁多いか？（後続調整の柔軟性）
  • 冗長な計算・記憶領域を避けているか？（スケーラビリティ）
  • タスクの大きさは同程度か？（均等な負荷配分）
  • 問題規模の増大に応じてタスク数がスケールするか？
  • 複数の代替分割案（domain / functional）を検討したか？

Communication 通信の目的

  • 分割で得たタスクは独立には実行できない。あるタスクの計算は別タスクのデータを必要とする。この情報フローを規定するのが通信ステージ。

① チャネル構造

データを必要とするタスク（consumer）と持つタスク（producer）を結ぶ。

② メッセージ

そのチャネル上で送受信される内容を specify する。

"The definition of a channel involves an intellectual cost and the sending of a message involves a physical cost. Hence, we avoid introducing unnecessary channels and communication operations."

Communication 通信パターンの4分類軸

軸意味例
Local / Global少数の隣接タスクとのみ / 多数のタスクと通信Jacobi 差分法（local）/ 並列総和リダクション（global）
Structured / Unstructured木・格子など規則的 / 任意グラフ有限差分法（structured）/ 有限要素法（unstructured）
Static / Dynamic通信相手が不変 / 実行時に変化格子（static）/ メッシュ再分割を伴う有限要素（dynamic）
Synchronous / Asynchronous協調して転送 / consumer が明示的に要求—（async）分散共有データ構造

4つの軸は "loosely orthogonal"（緩やかに直交）。実際の通信構造はこれらの組み合わせで記述される。

Communication 設計チェックリスト

§2.3.5

  • すべてのタスクがほぼ同数の通信操作を行うか？（不均衡は nonscalable の兆候）
  • 各タスクは少数の隣接タスクとのみ通信するか？（局所性）
  • 通信操作は並行して実行できるか？（divide-and-conquer で隠れた並行性を掘り起こす）
  • 異なるタスクの計算は並行して進められるか？（例：Gauss-Seidel → red-black への切替）

Agglomeration 凝集の目的

「抽象から具体へ」— P/C の決定を、実機で効率よく動くよう見直す。細粒度のままではタスク生成コスト・小メッセージの起動コストが過大。

3つの（時に相反する）ゴール：

① 通信コスト削減粒度を上げる
② 柔軟性の維持スケーラビリティ・移植性
③ SW工学コスト削減逐次コード再利用
"reducing communication costs by increasing computation and communication granularity, retaining flexibility with respect to scalability and mapping decisions, and reducing software engineering costs."

Agglomeration 主要概念

Granularity（粒度）

fine-grained な多数の小タスク → 結合して coarse-grained に。数を減らしサイズを大きくする。

Surface-to-volume 効果

通信量 ∝ 表面積、計算量 ∝ 体積。タスクを大きくすると通信/計算比が低下。高次元分割が有利。

Replication（複製）

原則は複製回避だが、通信を減らせるなら複製する価値あり。計算の複製（総和のbutterfly）/ データの複製（Fock行列）。

移植性のトレードオフ

"A program able to use only a fixed number of processors is a bad program." タスク数はパラメータ化し、プロセッサ数の1桁以上を保つ。

Agglomeration 数値例：表面対体積効果

5点ステンシルの 8×8 格子（2次元差分法）

設計タスク数通信回数転送データ量
1格子点＝1タスク64256256
4×4ブロック＝1タスク41664

タスク内部の通信が消滅し、通信回数は 256 → 16、転送量は 256 → 64 に激減。
体積（計算）は保ったまま、表面（通信）だけを大きく削減できる。

Agglomeration 設計チェックリスト

§2.4.4（抜粋）

  • 局所性の向上で通信コストを削減できたか？
  • 計算を複製したなら、利益がコストを上回ることを問題規模・プロセッサ数の範囲で検証したか？
  • データを複製したなら、スケーラビリティを損なっていないか？
  • 計算コストと通信コストが同程度のタスクが得られたか？
  • タスク数は問題規模に応じてスケールし続けるか？
  • 十分な並行性が現在・将来の対象計算機に確保されているか？
  • 負荷不均衡を招かず、タスク数をさらに減らせないか？
  • 逐次プログラム並列化時、コード変更のコストを考慮したか？

Mapping 割当の目的

各タスクを物理プロセッサに割り当てる最終ステージ。2つの相反する目標を満たそうとする。

並列性の最大化

同時実行可能なタスクを異なるプロセッサに配置。

通信コストの最小化

頻繁に通信するタスクを同じプロセッサに配置。

一般解は NP完全 → ヒューリスティック（負荷分散・タスクスケジューリング）に頼る。
Agglomeration で「1プロセッサ1タスク」まで統合済みなら、Mapping は A に吸収される。

Mapping 負荷分散とスケジューリング

負荷分散アルゴリズム

ドメイン分解＋計算量が可変/非構造な場合。

  • Recursive bisection（再帰二分割）
  • Local algorithms（近隣情報のみ、動的向き）
  • Probabilistic / Cyclic mappings

タスクスケジューリング

機能分解＋短命な多数タスクの場合。

  • Manager/Worker
  • Hierarchical Manager/Worker
  • Decentralized（分散型）

マッピングが自明：タスク数固定・均等・構造化通信のとき。動的負荷分散：実行時に計算量・通信量が変化する最も複雑なとき。

Mapping 設計チェックリスト

§2.5.3

  • SPMD設計なら、動的タスク生成に基づく案も検討したか？（逆も然り）
  • 集中型負荷分散なら、マネージャがボトルネックにならないか？（本体でなくポインタを渡す）
  • 動的負荷分散なら、異なる戦略の相対コストを実装コスト込みで評価したか？
  • 確率的・巡回手法なら、負荷分散に十分なタスク数があるか？（目安：プロセッサ数の10倍）

このチェックリストで PCAM は完結。ただしコード作成前に性能分析で設計を検証すべき。

まとめ

  • PC 前半は機種非依存で並列性とスケーラビリティを最大限に暴き出す。
  • AM 後半は機種依存で局所性・効率・負荷分散を作り込む。
  • 各ステージにチェックリストがあり、次に進む前に設計の欠陥を検出する。
  • 抽象→具体の分離により、ハードウェアが変わってもアルゴリズムを書き直さず調整できる。
