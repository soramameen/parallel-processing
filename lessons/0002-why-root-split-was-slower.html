なぜ root-split 並列版は理想通りにスケールしなかったか

Lesson 0002 · 前提: Lesson 1 — root-split 並列 CLIQUES の仕組み

soc-sign-epinions（131,828頂点・711,210辺）で 1/2/4/8 worker を実測したところ、 こうなった。

worker数時間対逐次
1（逐次）184.7 秒1.00×
2240.2 秒0.77×
4236.3 秒0.78×
8172.9 秒1.07×

件数（22,226,173）と最大クリークサイズ（94）はすべての条件で一致していたので、 正しさは壊れていない。問題は純粋に速度で、しかも非単調： 2・4 worker では逐次版より遅くなり、8 worker でようやく逐次版をわずかに上回る（それでも 1.07倍——8倍の理想には遠く及ばない）。Lesson 1 で「アルゴリズムを変えずに枝を並列化するだけ」 と説明した設計が、なぜここで裏目に出たのかを見る。

1. 分岐の数を数えてみる

_root_branches(graph) が作る枝の数、つまり ext = cand - Γ(pivot) の サイズを実測すると： 

root branches (|ext|): 128,270   # 全131,828頂点のほぼ全て
largest 5 branch cand sizes: [3558, 1841, 1819, 1381, 1289]
smallest 5: [0, 0, 0, 0, 0]

疎なグラフ（平均次数 約10.8）では、pivot の隣接頂点はごく一部にしかならない。つまり ext = cand - Γ(pivot) は「ほぼ全頂点」になる。Lesson 1 で書いた 「ext の各要素が1つの独立した枝になる」という性質は正しいが、 疎グラフでは枝の数がそのまま頂点数に近づくという帰結を見落としていた。 128,270個の枝のうち、サイズ0（何もしないで終わる）ものも多く含まれる—— 負荷はきわめて不均一である。

2. 「並列化できない直列区間」がある

さらに重要な発見がある。128,270個の枝それぞれの初期状態 (subg, cand) を 逐次版と同じ順序で作る前処理（_root_branches）は、 メインプロセスの中で1本のループとして実行される。これ自体を計測すると： 

root-branch split: 70.16 秒

これは何 worker を使っても消えない。Lesson 1 の Q1 で確認した通り、 cand -= {p} の順序を正しく再現しないと重複が生まれるため、 この前処理を並列化しようとすると設計そのものを変える必要がある。

Gustafson (1988) の議論を借りると、これは古典的な Amdahl の法則の状況に近い： 処理全体のうち並列化できない部分（ここでは70秒の逐次分割）が下限として残り、 worker 数をいくら増やしてもそれ以下にはならない。
Q1. worker を2→8に増やしたとき、理論上の実行時間の下限に最も近いのはどれ？

3. Amdahl 予測とのズレ、そして8 workerで急に速くなった謎

70秒の直列区間＋残り(184.7−70)秒を p 分割、という単純な Amdahl 予測 time(p) = 70 + 114.5/p と実測を比べると：

worker数実測Amdahl予測説明できない超過
2240.2秒127.4秒+112.8秒
4236.3秒98.8秒+137.5秒
8172.9秒84.5秒+88.4秒

どの worker 数でも実測は予測よりかなり悪い。しかも超過分は worker数に比例して増減していない（2→4で減るどころか増え、8で急に縮む）。 「worker起動コストが worker数に比例して重くなる」という単純な説明だとこの形にならない。 より筋が通るのは次の仮説（未検証・次の測定候補）：

  • タスク送信オーバーヘッドは worker数に依存しない固定コスト： ProcessPoolExecutor.map は既定で chunksize=1。128,270個のタスクを メインプロセスが1個ずつ pickle してキューに積む処理は、worker が何台あっても メインプロセス側の作業量は変わらない。計算側（ワーカーの実処理）が worker 増加で速く終わる ほど、この一定コストの割合が相対的に大きく見える——これは 2・4 worker で遅く、8 worker で 急に改善した非単調な形と整合する。
  • ワーカー起動コスト：macOS の既定 start method は spawn。 ワーカーごとにグラフ全体（131,828頂点）を initializer 経由で再構築する必要があり、 worker数だけこのコストが重複する（ただしこちらは worker数に比例するはずで、上の非単調な パターンを単独では説明しにくい）。
これは docs/plans/cliques-parallelization-ideas.md が候補2の限界として最初から 指摘していた内容（「枝の計算量の偏り、spawn での graph 複製が主な問題」）の実測による裏付けに なっている。設計段階の懸念が、実測で具体的な数字（70秒・128,270枝）として確認できた。

4. この結果の使いどころ

これは失敗ではなく、卒論として価値のある「うまくいかない例」である。 MISSION.md の該当項目：「なぜ理想的な線形スピードアップに ならないのか」を考察できる、という達成目標にそのまま対応する。次に検証すべき仮説は：

  • chunksize を大きくする、または枝を事前にバッチ化してタスク数自体を減らす （設計docの候補1が想定している方向）。
  • 枝サイズでソートしてから重い枝を先に投入する（負荷の偏りへの対処）。
一次資料 Gustafson (1988), "Reevaluating Amdahl's Law", CACM 31(5) — 固定サイズ問題での並列化限界（Amdahl）と、規模を広げたときの見え方 （Gustafson）の対比。「なぜ直列区間がボトルネックになるか」を理論的に位置づけるときの定番引用。 詳しくは RESOURCES.md を参照。
「70秒の前処理はどうやったら並列化できるか」「chunksize を変えると本当に速くなるか」は どちらも次の実験で確かめられる問いです。試してみたい方向があれば、エージェントに相談してから 着手してください。
