root-split 並列 CLIQUES の仕組み

Lesson 0001 · Mission: Bron–Kerbosch / Tomita CLIQUES の並列化

このリポジトリには現在、逐次版の Tomita CLIQUES （src/parallel_processing/cliques.py）と、それを並列化した root-split 版（src/parallel_processing/cliques_parallel.py）の 2つがある。このレッスンでは、後者が「何を」「なぜその方法で」並列化しているかを説明する。

1. 出発点：逐次版の探索木

逐次版の中心は expand(subg, cand) という再帰関数だった。 

def expand(subg: set[int], cand: set[int]) -> None:
    if not subg:
        result.append(frozenset(q))   # Q が極大クリーク
        return
    pivot = max(subg, key=lambda u: len(cand & graph.get(u, set())))
    ext = cand - graph.get(pivot, set())
    for p in list(ext):
        neighbours = graph.get(p, set())
        q.append(p)
        expand(subg & neighbours, cand & neighbours)
        q.pop()
        cand = cand - {p}   # ← p を処理し終えたら候補から除く

最初の呼び出しは expand(V, V)（V は全頂点集合）。この最初の呼び出しの中で 計算される ext（= CAND \ Γ(pivot)）の各要素 p が、探索木の 「最初の分岐」にあたる。

ext の中身は「pivot の隣接ではない候補頂点」。pivot 自身も pivot の隣接には含まれない ので、ext は空にならない（自己ループなしのグラフでは pivot ∈ ext）。

2. なぜ分岐ごとに独立して並列化できるのか

for p in ext のループの中身をよく見ると、各 p は自分の枝の中だけ で完結した再帰 expand(subg & N(p), cand & N(p)) を行い、見つかったクリークはすべて p を含む。つまり「頂点 p を含む極大クリーク」はすべてこの1つの枝の中に 閉じている。枝 p と枝 p'（p ≠ p'）が同じクリークを重複して 見つけることはない——これが並列化の土台になる性質である。

だから「ext の各要素を独立プロセスに配って、それぞれの枝を並列に探索し、最後に 件数を集計するだけでよい」というのが root-split の発想である。 アルゴリズムの中身（pivot 選択やビットセット化など）は一切変えていない、という点が重要。 docs/plans/cliques-parallelization-ideas.md の候補2にあたる、いちばん素朴な並列化。

3. 落とし穴：cand -= {p} の順序を再現する必要がある

逐次版のループをもう一度見ると、兄弟 p を処理し終えるたびに cand = cand - {p} をしている。つまり2番目の兄弟が使う cand は、 1番目の兄弟をすでに除いたもの。もしすべての枝に同じ cand（＝最初の ext と 同じ集合）を渡してしまうと、同じ極大クリークが複数の枝で重複してカウントされる。

そのため cliques_parallel.py の _root_branches は、 プロセスに配る前に、メインプロセス側で逐次版と同じ順序で cand を1つずつ削りながら、各枝の初期状態 (subg, cand) をあらかじめ 計算しておく。 

def _root_branches(graph: Graph) -> list[tuple[set[int], set[int]]]:
    subg = set(graph)
    cand = set(graph)
    pivot = max(subg, key=lambda u: len(cand & graph.get(u, set())))
    ext = cand - graph.get(pivot, set())
    branches = []
    for p in ext:
        neighbours = graph.get(p, set())
        branches.append((subg & neighbours, cand & neighbours))
        cand = cand - {p}   # 逐次版と同じ順序で削る
    return branches
Q1. もし _root_branches が cand -= {p} を省略し、 全ての枝に同じ cand（最初の ext 全体）を渡したら何が起きる？

4. なぜ共有カウンタを使わないのか

soc-sign-epinions（SNAP データセット）では極大クリークが約2,226万個見つかる。 もし全ワーカーが1つの共有カウンタ（例えば multiprocessing.Value）をロックしながら インクリメントしていたら、2,226万回のロック争奪がボトルネックになり、並列化の効果が消える。

そこで各ワーカー（_branch_task）は自分の枝の中で見つけた件数と最大クリークサイズを (local_count, local_largest) としてローカルに集計し、 親プロセスが ProcessPoolExecutor.map の結果を sum/max で まとめる。通信はワーカーごとに1回だけ。 

with ProcessPoolExecutor(
    max_workers=workers, initializer=_init_worker, initargs=(graph,)
) as executor:
    results = list(executor.map(_branch_task, subgs, cands))

count = sum(c for c, _ in results)
largest = max((size for _, size in results), default=0)
initializer=_init_worker でグラフを各ワーカープロセスの起動時に1回だけ渡している。 こうしないと、タスクを送るたびにグラフ全体を pickle して送ることになり、通信コストが跳ね上がる。

5. 現状の位置づけ（次のレッスンへの布石）

今のところ確認できているのは「正しさ」だけである：小規模グラフとランダムグラフで count_cliques_parallel_root の結果が逐次版と一致することを tests/test_cliques_parallel.py で検証済み。soc-sign-epinions での 逐次ベースラインは 184.7秒、極大クリーク数 22,226,173 （論文 Table 1 の期待値と一致）。1/2/4/8 ワーカーでの実測スピードアップは計測中で、 次のレッスンで扱う。

先取りして考えておくとよい問い：ext の各枝は本当に同じくらいの計算量か？ 違うとしたら、静的に「枝を1個ずつプロセスに渡す」やり方はどう困るか。この問いは docs/plans/cliques-parallelization-ideas.md の負荷偏り実測（最重1タスクが全体の 0.8%）とつながっている。

一次資料 Tomita, Tanaka, Takahashi (2006), "The worst-case time complexity for generating all maximal cliques and computational experiments", TCS 363(1) — pivot 選択の原論文。まずここを読むと、なぜ ext = cand - Γ(pivot) という削り方が 最悪計算量を最適にするのかがわかる。詳しくは RESOURCES.md も参照。
