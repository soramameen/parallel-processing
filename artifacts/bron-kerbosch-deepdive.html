01何を解くのか

「友達関係の中で、全員が互いに友達であるようなグループ」を、もう大きくできないところまで探す。

クリーク（clique）

頂点の部分集合で、どの2頂点も辺で結ばれているもの。「全員が互いに知り合い」なグループ。

極大クリーク（maximal clique）

これ以上どの頂点を追加してもクリークでなくなってしまうクリーク。つまり「これ以上大きくできないクリーク」。Bron–Kerbosch 法は、グラフ中の極大クリークを“すべて”列挙するアルゴリズムです。

注意: 「最大クリーク（maximum clique = 一番大きいもの1つ）」ではありません。極大クリークはグラフ全体で複数ありえます。私たちのコードはそれらをすべて返します。

今回のコードが扱うサンプルグラフ（頂点 1–6）

このグラフの極大クリークは {1,2,5}, {2,3}, {3,4}, {4,5}, {4,6} の5つ。これを目標の答えとして、アルゴリズムがどう導き出すかを見ていきます。

02核心: 3つの集合 R / P / X

Bron–Kerbosch のすべては、3つの集合を持ち回る再帰に帰着します。これらを理解すれば、あとは“枠組み”です。

R — 現在組んでいるクリーク P — 候補（R に追加してもクリークのままの頂点） X — すでに処理済み（もう追加しない）

R は現在注目しているクリークそのもの。P は「ここから1頂点選んで R に足せる」候補。X は過去に選んで R から外した頂点で、同じクリークを重複して出力しないための「除外」です。

出力のタイミング: 再帰のある呼び出しで P も X も空 になったら、そのときの R が極大クリークとして確定します。
「候補も無く、除外も無い」＝「これ以上大きくできないし、過去にも出力していない」＝極大クリーク、という理屈です。

各再帰ステップでは、P から頂点 v を1つ選びます。頂点 v をクリークに足すには、次の候補は「v の隣接頂点だけ」に絞られます（v と繋がっていない頂点はクリークになれないから）。なので新しい P, X はそれぞれ元の集合と N(v)（v の近傍）の積集合になります。

再帰の骨格（共通）

def bk(R, P, X):
    if not P and not X:
        出力(R)              # 極大クリーク確定
        return
    for v in P:               # 候補から1つ選ぶ
        bk(R ∪ {v}, P ∩ N(v), X ∩ N(v))
        P ← P − {v}         # v を候補から外し…
        X ← X ∪ {v}         # 除外に回す（重複防止）

枝刈りなしの素直な実装。上の骨格をほぼそのまま Python に書き下ろしたものです。

src/parallel_processing/bron_kerbosch.py（bron_kerbosch_simple）



    
読み方のポイント

    
    
      
  • cliques: list[frozenset[int]] = [] — 見つかった極大クリークを詰める箱。内側の recurse がこの箱に追記します（クロージャ）。

  • for v in list(p) — p を直接回すとループ中に書き換えてしまうので、list(p) でコピーを固定して回します。ここが初心者がよくハマる罠。
    • 
      
  • recurse(r | {v}, p & neighbours, x & neighbours) — まさに骨格の R∪{v}, P∩N(v), X∩N(v)。| は和集合、& は積集合。

  • p = p - {v}; x = x | {v} — 頂点 v を処理し終えたら P から外して X に移動。これが重複出力を防ぐ心臓部。
    • 
      
  • graph.get(v, set()) — v が隣接を持たない（孤立点）場合も安全に空集合を返す小さな工夫。

05pivot 版 — 無駄な分岐を刈る


    
simple 版の無駄

    
simple 版は P の頂点すべてを分岐に使いますが、実は「必ず失敗する分岐」がたくさん含まれます。極大クリークは「これ以上大きくできない」クリークです。もし今の状態で P の中にあるクリークの候補が残っているのに、そこから1頂点だけ選んで R を完成させると、「本当は P の別の頂点を足せたはず」なのに極大として出力してしまう……と見えて、実はそうはなりません。代わりに同じ極大クリークを何度も（P の別の頂点経由で）出力しようとして時間を浪費します。


    
pivot のアイデア

    
P ∪ X から1つ ピボット u を選びます。すると極大クリークは「u を含むか、u の隣接でない頂点を含むか」のどちらかしかありません。つまり P ∩ N(u) に属する頂点だけで分岐を作れば十分で、それらは「u を含む場合」でカバーされます。


    

      
結論: P − N(u) の頂点だけを分岐に使えば、同じ極大クリーク集合をすべて見つけつつ、再帰呼び出しの数を大幅に減らせます。

      どの u を選んでも正しいですが、P の中で近傍が最も多い頂点を贪欲に選ぶと P − N(u) が最小になり、枝刈りが最大化します。

    


    
bron_kerbosch_pivot（simple 版との“差分”に注目）




    
simple 版との差は2行だけ

    
    
      
  • pivot = max(p | x, key=lambda v: len(graph.get(v,set()) & p)) — P ∪ X から「P 内の近傍数が最大」のピボットを贪欲選択。
    • 
      
  • for v in list(p - graph.get(pivot, set())): — P − N(u) だけを分岐に使う。これが枝刈りそのもの。
    • 
    

    
それ以外は simple 版と同一です。出力する極大クリークの集合は変わりません（次節のトレースとテストで一致を確認済み）。

  







  

    
06計算量と効果

    
頂点数 n、辺数 m、グラフ中の極大クリーク数を μ とします。最悪ケースの極大クリーク数は μ = Θ(3^(n/3))（Moon–Moser）なので、列挙自体がこれだけ必要です。


    

      
バリアント最悪計算量実感

      
        
simple（枝刈りなし）O(3^(n/3)) だが定数倍が大きく、密なグラフで無駄呼び出しが爆発小さいグラフなら十分。分かりやすい。

        
pivotO(3^(n/3))・定数倍で有利、各再帰の分岐数を実質最小化実用上の標準。実質的に“これ”が Bron–Kerbosch。

        
（発展）退行あり pivotO(3^(n/3))・メモリ O(n) で 最悪最適巨大グラフ向け。今回は実装せず。

      
    


    

      
両者は同じ答えを出す: プロジェクトの tests/test_bron_kerbosch.py は、手計算の期待値に両者が一致することと、乱択グラフ50個で simple 版と pivot 版の出力が完全一致することを検証しています。pivot は“速くなるだけ”で正しさは変わりません。
